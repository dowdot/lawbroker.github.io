CORS跨域資源共享

Posted on 2019-10-17 | In Tools

今天從公司COPY了一份純HTML檔案的專案在自己的電腦上作業，單純用來做簡單的Prototype Layout用。
沒想到，在公司的 HTTP Server 上執行是沒問題的，但在自己本地端的Chrome瀏覽器上就是無法執行。

上網一查才知道原來是XMLHttpRequest 因為安全性的考量只允許相同域名的請求，而CROS(Cross-Origin Resource Sharing)的請支援的協議有：http,data,chrome,chrome-extension,https，但用檔案總管開啟檔案在瀏覽器預設跑的協定卻是 file:// 所以就被拒絕了!!

解決的方式大約有以下幾種：

增加 Chrome 執行參數

  1. 在CMD mode 下，cd 到chrome.exe 所在的資料夾，加參數執行Chrome
    (需先關閉所有的chrome)

  2. 在檔案總管，找到 chrome.exe 按右鍵->內容 修改目標屬性，同樣加上 –allow-file-access-from-files　參數
    (需先關閉所有的chrome)

架本地端的Web Server

這個做法有很多種，其中最直覺的方式是:

  1. 裝 MAMP 之類Web Server
  2. 在VS Code 裡安裝 Live Server

結論:

我最後是在 VS Code裡安裝 Live Server這個套件，因為改瀏覽器參數，怕帶來安全上的風險。而只是為了短暫瀏覽HTML網頁裝MAMP就太麻煩了!!
既然有在用VS Code，安裝一個外掛相對簡單一些，而且用Live Server反而有助於前端開發。應該說，早該裝了!!

稽核員希望主機上的指令歷程可以保留的更長一點，最好還能夠被記錄在額外的日誌檔案中。
所以，就來實作一下吧！！

需求說明

  1. 指令歷程能被保留3000筆

  5. 考量日後上傳至ELK的彈性，日誌檔使用”,”做為欄位區隔符號
  6. 非root帳號時，指令歷程也能有權限寫入日誌檔
  7. 使用 history 指令顯示歷程時需顯示日期及時間
  8. 需能記錄執行指令時當下所以的路徑

Gmail API Overview

Posted on 2019-02-26 | In G Suite

API Overview

Gmail API 是以RESTful API的型式來供用來包含對Gmail信箱進行: 讀取和寄送郵件、管理草稿和附件、搜尋郵件使用標籤、推送通知及管理Gmail設定等。可以看成是Gmail版的IMAP協定，當然也可以利用程式擷取信件內容做特殊的應用。因為是Google 自家提供的服務，相容整合的功能不會是太大的問題，但缺點在對API呼叫的數量上有所限制。

Key resource types

Gmail API可對以下6個資源型態進行操作：

  • Messages - 郵件訊息
  • Labels - 標籤
  • Drafts - 草稿
  • History - 歷史紀錄
  • Threads - 會話群組
  • Settings - 信箱設定

其中 Messages and labels 是信箱的基本單位;而 Drafts(草稿)， history(歷史紀錄) 及 threads 都包含一個或多個額外的 metadata。

NGINX 設定檔產生器

Posted on 2019-02-18 | In Tools

大家都知道，人的年紀漸漸在大，就愈不容易改變長年以來的習慣；今天的主題就是這樣來的~~

Apache用了十幾年了，一直用的蠻習慣的。雖然知道NGINX也紅很多年了，但就沒特別的動機去嘗試!!

最近，一個PHP Lab環境用了LaraDock來跑Laravel，LaraDock是個Container環境，雖然裡面也包有Apache套件，但大部份使用的人習慣會用NGINX。所以，心裡想著，既然是玩個Lab，只是下個 docker-composer up -d NGINX，目標就是將Laravel跑起來就好了嘛！！

沒想到，Lab的功能出乎我原本的預期。漸漸的發現，要達成某些功能，我得去修改NGINX的設定檔才行。這下，可慘了~~　對NGINX好陌生丫!! 光要加個 Virtual Host並綁上SSL憑證，都不知該從何下手。

還好有NGINXconfig.io 這個好站，只要將需求在Web介面上設定好，它就會幫你產生設定檔。這對我這個NGINX新生給了很大的幫助~~

有了這麼好的工具，日後我應該會慢慢的將網頁伺服器改成NGINX了!!
